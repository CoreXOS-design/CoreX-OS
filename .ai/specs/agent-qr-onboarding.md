# Agent QR Onboarding — Spec

> Each agent gets a unique QR code. A prospect scans it in the CoreX mobile app, fills in their details, sets a password — and is instantly a Contact in the agent's agency, attributed to that agent, and signed into the Client Portal.
> Created: 2026-05-12
> Status: BUILT 2026-05-12

## Purpose
Eliminate the friction of "ask your agent to add you." Hand a prospect your card / scan a QR in person → they self-onboard → you get a new lead in seconds, already attributed.

## Pillars
- **Agent** (primary) — every QR is keyed to one `User` (agent role); the resulting Contact's `created_by_user_id` = that agent
- **Contact** (primary) — onboarding creates a new Contact row in the agent's agency
- **Agency** (secondary) — the new Contact is in the agent's `agency_id`; the new `ClientUser.created_by_agency_id` = that agency

## Non-negotiables compliance
- Auto-link to existing `ClientUser` via `ContactObserver` (built 2026-05-12) — if the scanning prospect already has a CoreX account, the new Contact links to it automatically
- Origin-agency rule applies — the agent's agency becomes the `created_by_agency_id`
- All endpoints under `/api/v1/client-auth/agent-qr/*`, named, discoverable via `/admin/api`
- No new permissions required — every agent automatically gets a QR

## Data model

### `users` — additions
| Column | Type | Purpose |
|---|---|---|
| `qr_code_slug` | string(16), nullable, **unique**, indexed | Stable short identifier embedded in the QR URL. Generated on User creation; backfilled for existing users. Base32 (Crockford-ish) lowercase, 10 chars. |
| `qr_reroute_user_id` | unsignedBigInteger, nullable, indexed | Reroute pointer. NULL = live agent, resolve normally. Set when an agent is deleted: scans of this agent's slug resolve through the pointer to the named live agent. The slug itself **never moves** — it stays on the original agent forever as the audit anchor. Added 2026-05-17, migration `2026_05_17_120000_add_qr_reroute_user_id_to_users.php`. |

Slug is generated by `User::ensureQrSlug()` — called lazily by the portal view and the public lookup endpoint. Migration backfills every existing user.

## QR resolution & rerouting (added 2026-05-17)

A scanned slug resolves via `User::resolveByQrSlug($slug)`, used by both `show` and `register`:

1. Look up the user owning the slug (no global scopes; ignore active/deleted at this step).
2. If that user is **active and not soft-deleted** → that's the agent. Return.
3. Otherwise, if they have a `qr_reroute_user_id` → hop to that user and repeat from step 2 (**chained** — if the target later leaves too, their own reroute is followed).
4. Loop guard (visited-set) and a dead-end (departed agent with no reroute) both return `null` → endpoints respond 404 as before.

**Rerouting is mandatory on every agent delete.** Every agent has a QR slug and printed cards/signage are in the wild, so the agent-delete modal *always* forces the admin to pick a QR reroute target (an active agent in the same agency) — even when the agent has zero attached properties/contacts. The picker defaults to the properties/contacts reassignment target but is independently selectable. The reroute pointer is written **before** the source agent is soft-deleted. Logged as `agent.qr_rerouted`. See `.ai/specs/agent-delete-reassignment.md`.

## QR encoding (updated 2026-06-10)
The QR encodes a canonical, human-readable public-profile URL:
```
https://corexos.co.za/corex/agents/{name-slug}/{qr_code_slug}
```
- `{name-slug}` is `Str::slug(agent name)` — cosmetic only. A rename triggers a
  301 to the canonical slug; it never resolves the agent.
- `{qr_code_slug}` is the stable, unique 10-char ID tag that resolves the agent
  (via `User::resolveByQrSlug()`, incl. the departed-agent reroute chain).
- Generated by `User::publicProfileUrl()` (= `qrCodeUrl()`), driven by
  `config('app.url')` so each environment emits its own host.
- **Web (no app):** the URL lands on the agent's **public profile page**
  (`corex.agents.public`, no auth) — the same standalone live-preview the agent
  sees internally (profile, listings, testimonials, articles). This replaces the
  old 404 behaviour.
- **Mobile app:** the scanner matches the `/corex/agents/{name}/{slug}` pattern,
  extracts the trailing 10-char slug, and routes to the onboarding screen using
  `GET /api/v1/client-auth/agent-qr/{slug}` for preview — the onboarding API is
  unchanged. **Mobile-app change required:** the URL matcher must accept the new
  pattern (was `/r/a/{slug}`). See `.ai/specs/agent-qr-MOBILE-PROMPT.md`.

### Backwards compatibility
The original `/r/a/{slug}` URL (printed on cards/signage already in the wild)
still works: a server route resolves the slug and 301s to the canonical public
profile (`agent.qr.legacy` → `corex.agents.public`).

## API surface — under `/api/v1/client-auth/agent-qr/*`

| Method | Route | Name | Auth | Purpose |
|---|---|---|---|---|
| GET  | `/api/v1/client-auth/agent-qr/{slug}` | `client-auth.agent-qr.show`     | none | Preview agent info for the onboarding screen. Returns `{agent: {first_name, last_name, full_name, photo_url, agency: {id, name, slug}}}`. 404 if slug unknown. |
| POST | `/api/v1/client-auth/agent-qr/{slug}/register` | `client-auth.agent-qr.register` | none | Body: `{first_name, last_name, phone, email, password, password_confirmation, device_name?}`. Creates Contact + ClientUser, returns `{token, contact, agency, agent}`. 409 if `ClientUser` already exists for that email (response includes `existing: true` and the agent-linked Contact still gets created/linked). Rate limit: 5/hour per slug per IP. |

## Flow (mobile)
```
Login screen → tap "Scan agent QR"
  → camera opens → reads https://…/r/a/{slug}
  → GET /agent-qr/{slug}     → shows "You're signing up with André Roets — Home Finders Coastal"
  → user fills: first name, surname, cell, email, password
  → POST /agent-qr/{slug}/register
       ├ success     → save sanctum token → land on Client home in that agency
       └ email taken → "You already have a CoreX account — sign in instead." (still creates the Contact linked to this agent so they show up as a new lead next time they log in)
```

## Acceptance criteria
1. Every existing User has a `qr_code_slug` after migration.
2. New Users get a slug auto-generated on first portal visit (lazy `ensureQrSlug`).
3. `GET /agent-qr/{slug}` returns agent + agency, no PII beyond name + photo.
4. `POST /agent-qr/{slug}/register` creates: a Contact in the agent's agency (with `created_by_user_id` = agent), a ClientUser, links them, returns a Sanctum `client`-ability token, and logs `password_set` + `password_login_success` to `client_access_logs`.
5. If email belongs to an existing ClientUser, the Contact is still created and linked to the existing ClientUser; response carries `existing: true` and **no new password is written**.
6. The QR appears at the bottom of the Profile tab and contains the canonical URL.
7. Rate limit 5/hour per slug+IP enforced on `register`.
8. `/admin/api` lists both new endpoints.
9. `php artisan view:clear && route:clear && cache:clear`, `php -l` clean, `scripts/dev-check.ps1` 0 new failures.
