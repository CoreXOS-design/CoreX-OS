## 6. The Execution Path (stale-worker class) — **HIGH RISK, LIVE**

`[LIVE]` `ps` at audit time shows **two** workers consuming the `/corex` `default` queue:
```
PID 2205624  Fri Jun 26 20:52:16  php /corex/artisan queue:work --queue=default,matching --sleep=3 --tries=3 --max-time=3600   [supervised: corex-worker-live]
PID 1584906  Thu Jun 25 19:21:09  php artisan queue:work --tries=3   [parent 1329360 = "PM2 v6.0.8 God Daemon (/root/.pm2)"]
```
- The **supervised** worker (`/etc/supervisor/conf.d/corex-worker.conf` → `corex-worker-live`) self-restarts hourly (`--max-time=3600`) so it picks up new code each hour. ✅
- The **PM2 worker PID 1584906** is an **un-supervised duplicate**: booted **Jun 25 19:21**, **~25.7h uptime**, **no `--max-time`** (never self-restarts), **no `--queue` filter** (defaults to `default`). A long-lived `queue:work` holds all PHP classes in memory for its entire life.
- Core P24 files were modified **after** that worker booted `[LIVE]`:
  - `Property24ListingMapper.php` — **2026-06-26 19:46** (suburb resolver, FEATURE_TAG_MAP, photo source)
  - `Property24ApiClient.php` — **2026-06-26 17:32**
  - `SubmitListingToProperty24.php` — **2026-06-26 17:32**
  - (latest commit: `cd6890c0` 2026-06-26 20:52)
- **Therefore PID 1584906 is executing the Jun-25 mapper/client.** Since both workers pull `default`, **whichever grabs a given `SubmitListingToProperty24` job decides whether it runs current or ~25-hour-old code** — non-deterministically. This is the exact stale-worker hazard, and it also feeds Bug #4 (two workers → concurrent saves of the same listing).

**opcache `[LIVE]`:** `opcache.enable=On`, **`opcache.enable_cli=Off`**, `validate_timestamps=On`, `revalidate_freq=2`. CLI/worker opcache is off, so a *fresh* `php artisan` invocation always reads current files — but that does **not** help a persistent worker, which never re-reads class files until the process restarts.

**Deploy requirement (on record):** every deploy that changes P24 code MUST restart **all** workers — `php artisan queue:restart` for the supervised pool **and** kill/restart the PM2 worker (PID 1584906 / PM2 daemon 1329360). The PM2 worker appears to be an unmanaged leftover and should be removed so only the supervised pool serves the queue.
